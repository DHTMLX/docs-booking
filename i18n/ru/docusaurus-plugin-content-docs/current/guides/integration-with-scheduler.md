# Интеграция с DHTMLX Scheduler {#integration-with-dhtmlx-scheduler}

Это руководство показывает, как интегрировать виджет DHTMLX Booking с [DHTMLX Scheduler](https://docs.dhtmlx.com/scheduler/). Интеграция преобразует события Scheduler в слоты Booking на стороне сервера.

## Основные концепции {#understand-the-main-concepts}

Интеграция строится на преобразовании событий Scheduler в слоты Booking. Перед началом работы учтите следующее.

**События Scheduler и слоты Booking.** Scheduler работает с событиями — как разовыми, так и повторяющимися. Booking генерирует доступные временные слоты из этих событий. [Пример ниже](#example) генерирует слоты бронирования из расписания врача путём преобразования JSON-данных на стороне сервера.

**Ограничения повторяющихся событий.** Booking поддерживает только еженедельные повторяющиеся события, заданные как `INTERVAL=1;FREQ=WEEKLY` в Scheduler. Scheduler поддерживает любые шаблоны повторения, поэтому ограничьте другие варианты повторения в настройках Scheduler.

**Обработка часовых поясов.** Booking интерпретирует временные метки в локальном часовом поясе. Если вы используете глобальные временные метки, преобразуйте их в локальный часовой пояс перед отправкой в Booking и обратно в UTC перед сохранением. Инструкции по преобразованию см. в разделе [Преобразование UTC-данных в локальный часовой пояс](guides/saving-reservations.md#convert-utc-data-to-the-local-timezone).

**Стратегии слотов Booking.** Выберите один из двух подходов для построения расписания:

- [`slots`](api/config/booking-data.md) и [`usedSlots`](api/config/booking-data.md) — строить расписание и исключать забронированные слоты (стратегия, рассматриваемая здесь)
- [`availableSlots`](api/config/booking-data.md) — явно перечислять доступные для бронирования слоты; подходит для событий без повторений

## Пример {#example}

Пример ниже интегрирует Booking со Scheduler путём преобразования расписаний врачей в слоты бронирования. Интеграция использует четыре конечные точки данных:

- `/doctors/worktime` — данные Scheduler (расписания врачей) с повторяющимися и разовыми событиями; источник временных слотов для Booking
- `/units` — итоговые слоты Booking, сгенерированные из данных `worktime` на стороне сервера; см. [пример бэкенда](https://github.com/DHTMLX/scheduler-booking-go)
- `/doctors/reservations` — вспомогательная коллекция, отображающая `usedSlots` в представлении временной шкалы; содержит уже забронированные слоты из формы Booking
- `/doctors` — все врачи с именами и идентификаторами; передаёт информацию о враче в оба виджета — Scheduler и Booking

Преобразование событий в слоты — основа интеграции. [Следующий раздел](#convert-scheduler-events-to-booking-slots) описывает правила преобразования.

## Преобразование событий Scheduler в слоты Booking {#convert-scheduler-events-to-booking-slots}

Правила ниже генерируют слоты бронирования из расписания врача на основе JSON-данных, преобразованных на стороне сервера. Во всех примерах используется расписание на следующий период: с 2025-03-13 по 2027-03-13.

### Правило 1. Создание слота из разового события {#rule-1-create-a-slot-from-a-single-event}

Для каждого разового события преобразуйте время начала и окончания в слот Booking. Добавьте запись в массив `slots` и включите дату события в массив `dates`.

Следующий фрагмент кода показывает разовое событие Scheduler:

~~~json
{
   "doctor_id": 1,
   "start_date": "2025-03-18 02:00:00",
   "end_date": "2025-03-18 06:00:00"
}
~~~

Следующий фрагмент кода показывает результирующий слот Booking:

~~~json
{
   "id": 1,
   "slotSize": 20,
   "slotGap": 5,
   "slots": [
     {
       "from": "02:00",
       "to": "06:00",
       "dates": [
         1742256000000 // 2025-03-18 00:00:00 (временная метка)
       ]
     }
   ]
}
~~~

### Правило 2. Преобразование повторяющегося события {#rule-2-convert-a-recurring-event}

Сопоставьте повторяющееся событие с еженедельным шаблоном. Даты начала и окончания повторяющегося события в Scheduler должны совпадать с датами [`start`](api/config/booking-start.md) и [`end`](api/config/booking-end.md) в Booking. В противном случае создайте заполнители для дат до и после повторяющегося события (см. [Правило 7](#rule-7-handle-events-that-start-after-the-booking-start-date)).

Следующий фрагмент кода показывает повторяющееся событие Scheduler, которое повторяется еженедельно в рабочие дни (с понедельника по пятницу):

~~~json
{
   "doctor_id": 1,
   "start_date": "2025-03-13 09:00:00",
   "end_date": "2027-03-13 00:00:00",
   "rrule": "INTERVAL=1;FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR",
   "duration": 28800
}
~~~

Booking представляет еженедельное расписание в виде единого правила с одинаковым временем начала и окончания для каждого рабочего дня.

Следующий фрагмент кода показывает результирующие слоты Booking:

~~~json
{
   "id": 1,
   "slotSize": 20,
   "slotGap": 5,
   "slots": [
     {
       "from": "09:00",
       "to": "17:00",
       "days": [1, 2, 3, 4, 5] // с понедельника по пятницу
     }
   ]
}
~~~

### Правило 3. Разбиение события, охватывающего несколько дней {#rule-3-split-an-event-that-spans-multiple-days}

Booking генерирует слоты в пределах одного дня. Если событие охватывает два дня (например, начинается в 20:00 и заканчивается в 04:00), разбейте его на два слота — по одному на каждый день.

Например, смена врача, начинающаяся в субботу вечером и заканчивающаяся в воскресенье утром, разбивается на два правила: одно для субботы и одно для воскресенья.

Следующий фрагмент кода показывает многодневное событие Scheduler:

~~~json
{
   "doctor_id": 2,
   "start_date": "2025-03-13 20:00:00",
   "end_date": "2027-03-13 00:00:00",
   "rrule": "INTERVAL=1;FREQ=WEEKLY;BYDAY=SA",
   "duration": 28800
}
~~~

Следующий фрагмент кода показывает два результирующих слота Booking — по одному на каждый день:

~~~json
{
   "id": 2,
   "slotSize": 45,
   "slotGap": 5,
   "slots": [
     {
       "from": "20:00",
       "to": "24:10",
       "days": [6] // суббота
     },
     {
       "from": "00:10",
       "to": "04:00",
       "days": [0] // воскресенье
     }
   ]
}
~~~

### Правило 4. Добавление разового события к повторяющемуся расписанию {#rule-4-add-a-single-event-to-a-recurring-schedule}

Если разовое событие расширяет повторяющееся расписание, генерируйте слоты для обоих. Добавьте даты разового события в массив `dates` правила повторяющегося события.

Этот пример объединяет два события Scheduler:

- повторяющееся событие — доступность врача с 09:00 до 17:00 в рабочие дни
- разовое событие — дополнительная доступность с 02:00 до 06:00 18 и 19 марта

Следующий фрагмент кода показывает оба события Scheduler:

~~~json
[
    // повторяющееся событие
   {
     "doctor_id": 1,
     "start_date": "2025-03-13 09:00:00",
     "end_date": "2027-03-13 00:00:00",
     "rrule": "INTERVAL=1;FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR",
     "duration": 28800
   },

   // разовые события
   {
     "doctor_id": 1,
     "start_date": "2025-03-18 02:00:00",
     "end_date": "2025-03-18 06:00:00"
   },
   {
     "doctor_id": 1,
     "start_date": "2025-03-19 02:00:00",
     "end_date": "2025-03-19 06:00:00"
   }
]
~~~

Booking объединяет повторяющееся событие и разовые события в одно правило. Даты разовых событий (18 и 19 марта) имеют более высокий приоритет и добавляются в массив `dates` правила повторяющегося события. О порядке приоритетов см. в разделе [Определение правил слотов](guides/configuration.md#define-slot-rules).

Следующий фрагмент кода показывает объединённые слоты Booking:

~~~json
{
   "id": 1,
   "slotSize": 20,
   "slotGap": 20,
   "slots": [
     {
       "from": "02:00",
       "to": "06:00",
       "dates": [
         1742256000000, // 2025-03-18 00:00:00
         1742342400000  // 2025-03-19 00:00:00
       ]
     },
     {
       "from": "09:00",
       "to": "17:00",
       "days": [1, 2, 3, 4, 5],
       "dates": [
         1742256000000, // 2025-03-18 00:00:00
         1742342400000  // 2025-03-19 00:00:00
       ]
     }
   ]
}
~~~

### Правило 5. Изменение отдельного экземпляра повторяющегося события {#rule-5-modify-a-single-instance-of-a-recurring-event}

Если отдельный экземпляр повторяющегося события изменяется (например, смещается время на одну дату), создайте новый слот с обновлённым временем. Добавьте дату в массив `dates`, который имеет приоритет над массивом `days` для этой даты.

Следующий фрагмент кода показывает повторяющееся событие и его изменённый экземпляр:

~~~json
[
   {
     "doctor_id": 1,
     "start_date": "2025-03-13 09:00:00",
     "end_date": "2027-03-13 00:00:00",
     "rrule": "INTERVAL=1;FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR",
     "duration": 28800
   },
   {
     "doctor_id": 1,
     "start_date": "2025-03-14 03:00:00",
     "end_date": "2025-03-14 11:00:00",
     "recurring_event_id": "1",
     "original_start": "2025-03-14 09:00"
   }
]
~~~

Следующий фрагмент кода показывает правило повторяющегося события и переопределение для изменённой даты:

~~~json
{
   "id": 1,
   "slotSize": 20,
   "slotGap": 5,
   "slots": [
     {
       "from": "09:00",
       "to": "17:00",
       "days": [1, 2, 3, 4, 5]
     },
     {
       "from": "03:00",
       "to": "11:00",
       "dates": [
         1741910400000 // 2025-03-14 03:00:00 (изменено)
       ]
     }
   ]
}
~~~

### Правило 6. Удаление отдельного экземпляра повторяющегося события {#rule-6-delete-a-single-instance-of-a-recurring-event}

При удалении отдельного вхождения из повторяющегося события отразите это в правилах Booking. Создайте правило для удалённой даты с пустым временным интервалом и свойством `dates`, которое имеет более высокий приоритет, чем `days`.

Следующий фрагмент кода показывает повторяющееся событие и его удалённое вхождение:

~~~json
[
   {
     "doctor_id": 5,
     "start_date": "2025-03-14 09:00:00",
     "end_date": "2027-03-13 00:00:00",
     "rrule": "INTERVAL=1;FREQ=WEEKLY;BYDAY=TH,FR,SA,SU",
     "duration": 28800
   },
   {
     "doctor_id": 5,
     "start_date": "2025-03-23 09:00:00",
     "end_date": "2025-03-23 17:00:00",
     "recurring_event_id": "15",
     "original_start": "2025-03-23 09:00",
     "deleted": true
   }
]
~~~

Следующий фрагмент кода показывает правило повторяющегося события и пустой интервал, исключающий удалённую дату:

~~~json
{
    "id": 5,
    "slotSize":60,
    "slotGap":10,
    "slots":[
        {
            "from": "09:00",
            "to": "17:00",
            "days": [4, 5, 6, 0] // с четверга по воскресенье
        },
        {
            "from": "00:00",
            "to": "00:00",
            "dates": [
                1742688000000 // 2025-03-23 00:00:00 (удалённое вхождение)
            ]
        }
    ]
}
~~~

### Правило 7. Обработка событий, начинающихся после даты начала Booking {#rule-7-handle-events-that-start-after-the-booking-start-date}

Если повторяющееся событие начинается после даты начала Booking (по умолчанию — сегодня, в этих примерах 2025-03-13), создайте правила с пустыми временными интервалами для дат до начала события. Это исключает такие даты из повторения.

Следующий фрагмент кода показывает повторяющееся событие, начинающееся на четыре дня позже даты начала Booking:

~~~json
{
    "doctor_id": 5,
    "start_date": "2025-03-17 09:00:00",
    "end_date": "2027-03-13 00:00:00",
    "rrule": "INTERVAL=1;FREQ=WEEKLY;BYDAY=SU,MO,TU,WE,TH,FR,SA",
    "duration": 28800
}
~~~

Следующий фрагмент кода показывает правило повторяющегося события и пустые интервалы для четырёх дат до начала события:

~~~json
{
    "id": 5,
    "slotSize":60,
    "slotGap":10,
    "slots": [
        { "from": "09:00", "to": "17:00", "days": [0, 1, 2, 3, 4, 5, 6] },
        { "from": "00:00", "to": "00:00", "dates": [
            1741820400000,  // 13 марта 2025
            1741906800000,  // 14 марта 2025
            1741993200000,  // 15 марта 2025
            1742079600000   // 16 марта 2025
        ]}
    ]
}
~~~
